1) Qué es el producto

LeadSignal es una capa de inteligencia comercial para inmobiliarias. No está pensado como “otro CRM” ni como “chatbot genérico”: su propuesta central es detectar qué leads realmente importan, recomendar la próxima acción y mostrar dónde se están perdiendo oportunidades.

Superficie actual del producto: /dashboard, /leads, /leads/[id], /opportunities, /insights, /properties, /team, /settings, más landing comercial en /.

2) Stack completo y arquitectura

  • Frontend: Next.js App Router, React 19, Tailwind v4, componentes reutilizables.
  • Backend app: Route Handlers en app/api/** + Server Actions para operaciones de leads.
  • Autenticación: better-auth con sesión en DB y requireSessionContext().
  • Multitenancy: scoping por agencyId derivado de sesión (no por query libre).
  • Persistencia: PostgreSQL + Prisma (schema robusto de operación comercial e inteligencia).
  • Integración de canales: Webhook Meta en /api/webhooks/meta + salida por Graph API.
  • Testing: Vitest (unit/integration) + Playwright E2E.

5) Cómo funciona la parte de IA hoy

Sí hay sistema de inteligencia funcionando. Hoy opera en modo híbrido: reglas determinísticas como base + apoyo generativo opcional para ambigüedad.

  1. Ingreso de lead/mensajes por API o webhook Meta.
  2. Extracción de señales por reglas robustas (incluye presupuesto ARS/USD en múltiples formatos).
  3. Si hay ambigüedad crítica, fallback opcional a LLM con salida JSON estricta y guardrails.
  4. Scoring ponderado (0..100) + cálculo de urgencia/seriedad/prioridad.
  5. Recomendación de next best action según modo de política (conservador/agresivo).
  6. Reporte de confianza + detección de casos para revisión humana.
  7. Persistencia de outputs en Lead, LeadProfile, AiRun, AnalyticsEvent.
  8. Recalificación automática cuando entran nuevos mensajes al hilo.
  9. Evaluación online de calidad de calificación (evento qualification.eval.online).

Lógica de calificación (resumen práctico)

  • Señales fuertes: presupuesto claro, zona definida, timeline, tipo de propiedad, profundidad conversacional.
  • Urgencia: timeline corto y lenguaje de prioridad elevan score operativo.
  • Seriedad: cantidad/calidad de señales + consistencia en mensajes.
  • Objeciones penalizan probabilidad de cierre.
  • Política de agencia ajusta umbral de urgencia y estrategia de recomendación.
  • Guardrails duros para evitar salidas inválidas (rangos de timeline, dormitorios y presupuesto).

6) Qué tenemos vs qué falta

Ya tenemos (real hoy)

  • Auth + sesión + scoping tenant en rutas críticas.
  • RBAC fino por permisos (lectura/escritura/configuración/backoffice/jobs/billing).
  • CRUD operativo de acciones clave sobre leads.
  • Persistencia completa en PostgreSQL.
  • Integración Meta inbound/outbound funcional para MVP.
  • Cola crítica con reintentos y backoff para jobs de envío.
  • Auditoría de acciones sensibles (team/settings/channels/messages/billing).
  • Backoffice operativo con plan, jobs y auditoría.
  • SLO básico y telemetría de requests críticos.
  • Suite de tests unit, integration y E2E en camino crítico.

Pendiente / siguiente nivel

  • Alertas activas (Slack/Email/Pager) ante degradación de SLO y jobs fallidos.
  • Dead-letter queue dedicada y replay guiado por UI.
  • Trazabilidad distribuida completa por traceId en todos los endpoints.
  • Evals offline más amplias con dataset etiquetado por negocio.
  • Self-service billing completo (checkout, facturas, cobros).

7) Flujo completo para probar como usuario real

Esta sección es un guion QA completo para validar producto en orden natural de uso. Incluye setup, operación diaria, mensajería real por WhatsApp/Instagram, inteligencia, coordinación de equipo, analítica, observabilidad y backoffice.

Fase 0 — Preparación del entorno (una sola vez)

  1. Instalar dependencias: npm install
  2. Configurar .env desde .env.example (DB, auth, secretos Meta, cifrado).
  3. Ejecutar migraciones: npm run db:migrate
  4. Crear usuario admin de tenant nuevo: npm run create-account -- --new-agency --email admin@agencia.com --password 'TuPass123!' --name 'Admin Agencia'
  5. Levantar app: npm run dev

Resultado esperado: login operativo, tenant aislado y sin contaminación de datos demo.

Fase 1 — Recorrido comercial de onboarding (landing + acceso)

  1. Abrir / y recorrer narrativa completa (hero, problema, propuesta, proof, CTA).
  2. Verificar navegación interna (anchors y consistencia desktop/mobile).
  3. Entrar en /sign-in y autenticar con el usuario creado.

Resultado esperado: el usuario entiende valor, entra sin fricción y aterriza en el app shell.

Fase 2 — Configuración operativa base en /settings

  1. En “WhatsApp e Instagram (Meta)” crear conexión WhatsApp (phone_number_id) y conexión Instagram (account id).
  2. Cargar token cifrado y validar “Asistente de configuración” hasta checklist completo.
  3. Ejecutar “Prueba de envío” en ambos canales.
  4. Configurar preferencias IA:
    • Umbral de urgencia
    • Modo de matching (conservador/agresivo)
    • Tono de outreach
  5. En “Equipo activo”, crear al menos 2 agentes y editar roles (admin/agent).

Resultado esperado: conexiones activas, envío de prueba confirmado y equipo operativo en DB.

Fase 3 — Ingreso real de leads por conversación (WhatsApp + Instagram)

Objetivo: validar intake natural desde canales reales y calidad de extracción/señales.

  1. Enviar mensaje inbound por WhatsApp con contenido explícito: Hola, busco depto en Palermo o Belgrano. Presupuesto U$S 240k a 280k. Necesito mudarme en 2 meses.
  2. Enviar segundo mensaje WhatsApp con objeción: Me preocupan expensas altas y si hay cochera.
  3. Enviar mensaje inbound por Instagram: Busco casa en zona norte, presupuesto $220.000.000, podría cerrar en 3 meses.

Resultado esperado: creación/actualización de lead, conversación asociada, detección de presupuesto ARS/USD, zonas, timeline y objeciones.

Fase 4 — Operación diaria en /leads y /leads/[id]

  1. Confirmar que los leads aparecen en bandeja con score/prioridad/estado coherentes.
  2. Click en fila: cambia contexto del panel derecho sin salir de /leads.
  3. Click en flecha: abre dossier completo en /leads/[id].
  4. Ejecutar acciones operativas por lead:
    • Cambiar etapa
    • Asignar owner
    • Crear tarea
    • Completar/reabrir tarea
    • Agregar nota
    • Marcar visita agendada

Resultado esperado: persistencia inmediata en DB, actualización de vistas y coherencia en historial de etapas/tareas.

Fase 5 — Mensajería outbound editorial (WhatsApp/Instagram)

  1. En /leads, crear borrador outbound para un lead.
  2. Aprobar borrador (acción editorial humana).
  3. Confirmar encolado del envío (job crítico) + intento inmediato best-effort.
  4. Verificar estado del mensaje:
    • NOT_SENT si quedó en retry
    • DELIVERED si Meta aceptó envío
  5. Descartar un borrador alternativo y validar estado REJECTED.

Mensajes sugeridos para validar tono IA: Tenemos 2 opciones alineadas a tu búsqueda en Palermo, ¿te comparto una selección curada hoy?

Fase 6 — Comando operativo en /team

  1. Revisar follow-ups urgentes y leads sin owner.
  2. Confirmar que leads en riesgo reflejan cambios reales de etapa/actividad.
  3. Validar “visitas del día” luego de marcar visitas en dossier.
  4. Corroborar carga por miembro y tareas abiertas por owner.

Resultado esperado: la pantalla sirve para decidir trabajo hoy, no solo para mostrar estética.

Fase 7 — Inteligencia estratégica en /insights

  1. Verificar “high-intent lost leads”.
  2. Validar demora promedio de respuesta y leads sin follow-up.
  3. Comparar performance por fuente (WhatsApp vs Instagram vs otras).
  4. Comprobar conversión por etapa luego de operar varios leads.

Resultado esperado: señales accionables y trazables a datos reales, sin métricas literarias.

Fase 8 — Vistas ejecutivas: /dashboard, /opportunities, /properties, /analytics

  1. Recorrer tablero general y verificar consistencia de totales.
  2. En oportunidades, revisar priorización frente a score y urgencia.
  3. En propiedades, validar recomendaciones contra perfil de lead.
  4. En analítica, comprobar coherencia de tendencias con operaciones recientes.

Fase 9 — Backoffice real en /backoffice (solo admin)

  1. Auditoría: confirmar eventos por altas de equipo, cambios de settings y mensajes aprobados.
  2. Jobs: revisar cola crítica, reintentos y fallas.
  3. Billing/plan: editar plan/seat limit y validar persistencia.
  4. SLO: revisar success rate, error rate y p95 latencia 24h.

Resultado esperado: trazabilidad administrativa y operativa para gobernar el producto en producción.

Fase 10 — QA de seguridad y permisos (RBAC)

  1. Iniciar sesión como agente (no admin).
  2. Intentar acciones admin:
    • editar equipo
    • cambiar settings sensibles
    • acceder backoffice/billing/jobs
  3. Confirmar bloqueos con respuesta 403 FORBIDDEN.

Resultado esperado: separación clara de permisos y menor riesgo de operación accidental.

Fase 11 — QA automatizado y cierre

  1. Ejecutar unit/integration: npm test
  2. Ejecutar E2E: npm run test:e2e
  3. Documentar resultados y abrir issues solo sobre gaps reproducibles.

Criterio de salida recomendado: sin regresiones críticas en intake → operación → mensajería → insights → backoffice.

8) Testing automatizado actual

  • Unit: pipeline de calificación, seguridad Meta, parser payloads, utilidades y rutas de mensajes.
  • Integration: flujo lead intake→operación y mensajería Meta workflow.
  • E2E: journey principal de lead + aprobación/envío de mensaje Meta.

Comandos: npm test npm run test:e2e

9) Evaluación honesta: ¿“tenemos un sistema de IA”?

Sí, en el sentido operativo del MVP: hay un motor híbrido que extrae, puntúa, prioriza y recomienda acciones con persistencia, trazabilidad y fallback seguro.

No, si la definición exige una plataforma autónoma full-LLM con aprendizaje online continuo. Hoy la base es rule-based reforzada con LLM opcional en ambigüedad, que prioriza confiabilidad y costo para etapa MVP.

10) Prioridades recomendadas de mejora

  1. Alertas proactivas (SLO breach, jobs failed, canales Meta degradados).
  2. Dead-letter queue y replay administrable para incidentes de mensajería.
  3. Explicabilidad más rica en UI por cada recomendación IA (señales + peso).
  4. Forecast comercial con horizonte semanal/mensual en vistas ejecutivas.
  5. Billing self-service completo y políticas de planes más avanzadas.
